## What went well

- Understand-first, in parallel. Read the `modding` sources (`events.rs`,
  `registry.rs`), `03_modding` and `11_overload` directly while an Explore agent
  mapped the web-gallery / sounds / docs wiring at the same time. The ~730-line
  first draft then compiled with only three trivial warnings (an unused import
  and two unused constants) and zero Bevy-0.19 drift errors -- the standing
  "copy the shared UI/engine idioms verbatim" lesson applied, same win as the
  11_overload cycle.
- The end-to-end test was honest by construction. `fuel_rod_handler_runs_on_a_
  tick_through_the_bus` builds the handler from its shop JSON, spawns it, fires a
  `tick` through a live `GameEventsPlugin`, and asserts the world changed --
  driving the real registry -> queue -> action path, not the pieces. The
  reviewer specifically confirmed it is not faked. This is the recurring
  "exercise the real behaviour, reach for the hop" lesson (the last two retros)
  applied up front instead of under review pressure. Note the example is an
  external consumer of the crate, so `EventHandler::actions` is not reachable;
  the live-App test is the *only* honest way to drive the actions, and picking it
  deliberately avoided an aspirational-comment gap.
- The autopilot harness earned its keep twice in one cycle: first to verify the
  menu -> playing -> meltdown loop headlessly (the standing "an example is not
  done until it has driven its gameplay" gotcha), then again to re-verify the
  economy after the review's balance change. Removed before each commit; the
  forced-meltdown env toggle confirmed the game-over path too.
- Delegated an independent skeptical review to a subagent. It caught the "New
  best!"-on-ties bug and the heat-tension opt-out -- both things I had
  rationalised away in self-review. Third straight cycle where the outside
  skeptic finds what self-review misses.
- Caught a runtime-coverage gap myself: the autopilot drove the sim by calling
  `try_buy`/`fire` directly and the boot test only reached the menu, so the
  `Button`/`Interaction` click paths were never exercised. Rather than assume,
  read `bevy_ui`'s `Button` (`#[require(Interaction, ...)]`) to confirm the
  clicks are wired, and noted the keyboard fallback keeps it playable regardless.

## What went wrong

- Wrote an overclaim into the REVIEW.md response before the evidence backed it.
  The first draft of the R1.1 response said the uncapped-tier change made "every
  run eventually melt down"; the autopilot re-run immediately disproved it (a
  superhuman ~10-buys/second player holds equilibrium at tier 15+). Corrected to
  the honest "heat is now the permanent central concern; the trivial zero-heat
  infinite is gone, but masterful micro can still sustain." Root cause: I wrote
  the claim optimistically, then verified. This is the *exact* recurring lesson
  from the last two retros ("do not write a claim the evidence does not yet
  support"), and it bit a third time -- this time in a review response, not a
  test comment. The class is the same; only the surface moved.
- The heat-tension-is-opt-out design flaw shipped in the first implementation: a
  pure Solar + Market build made zero heat and scaled forever, so meltdown was
  effectively opt-in. Root cause: I tuned for "a good build stabilises" and never
  ran the adversarial question "what is the cheese strategy that dodges the core
  tension?" A one-pass self-critique of the economy before review would have
  caught it.
- LoC landed at ~1435, ~40% over the task's "~1000" guidance. Root cause: Bevy UI
  spawning (six shop cards with children, the heat bar, readouts) is verbose and
  I leaned into the crate's generous doc-comment convention without budgeting
  that verbosity against the stated cap. Defensible (it is the smallest game
  example) but a scope stretch I did not flag up front.

## What to improve next time

- Evidence before claim, everywhere -- not just test comments. Before writing any
  behavioural assertion (test comment, REVIEW response, docs sentence, TASK
  note), have the run/assertion that backs it in hand. Same rule the AGENTS.md
  testing-convention line already states; it just needs to generalise past tests.
- For any game/economy, run one explicit "what is the degenerate strategy?" pass
  before handing to review: name the build that dodges the intended tension and
  confirm it is not trivially optimal.
