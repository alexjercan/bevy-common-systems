## What went well

- Extending the existing `gen-placeholder-sounds.py` (rather than hand-authoring WAVs
  or reaching for ffmpeg/sox) kept the placeholder pipeline intact: `render_fx` is
  additive next to the original `render`, uses only the stdlib, and seeds its noise per
  file name so regenerating is byte-identical. That determinism is what let the review
  prove "only the six new files changed" via `git status` -- no spurious binary churn.
- Verified the right thing for audio: the headless autopilot fires the gun, so a clean
  run with no asset-load errors proves the `breach_*` files exist and are wired end to
  end. Timbre is a listen-test (headless has no audio out), and the review said so
  rather than overclaiming.
- Scoping menu_select / game_over as shared generic cues (not duplicated as breach_*)
  matched the crate's established reuse convention and avoided pointless files; the
  reasoning is captured in a code comment, the README and the docs note so a future
  session does not "fix" it by adding duplicates.

## What went wrong

- Nothing broke. Minor: I referenced `docs/2026-07-05-breach-fun-pass.md` in the AGENTS
  note before creating it; caught it in the same task and wrote the doc (a fun-pass
  overview covering all five gameplay tasks), so the link is not dangling. Worth doing
  deliberately -- a consolidated overview doc is more useful than five scattered notes.

## What to improve next time

- When a task edits a generator that also rewrites unrelated outputs, assert the diff is
  limited to the intended files (here: `git status` after regen) -- a non-deterministic
  generator would have rewritten all 13 existing sounds and buried the real change. Seed
  any randomness in asset generators for exactly this reason.
- Placeholder audio is "distinct and readable", not "final"; keep noise-vs-sweep-vs-sine
  as the cheap way to make categories audibly different without real sound design.
