# Spike: What would make 14_breach a fuller game (juice, combos, pickups, enemies)?

- DATE: 20260705-132024
- STATUS: RECOMMENDED
- TAGS: spike, example, fps, breach

## Question

The `14_breach` FPS example ships as a functional "survive the swarm" loop. The
ask is to make it *more of a game*: a richer main menu, better mobile, more juice
and dedicated sounds, a points-and-combos scoring layer, ground pickups
(HP/speed/fire-rate) and different enemy types. Which of these are real gaps
versus already-present, and how should each be scoped as a task that fits the
crate's "examples are the integration tests / harvest reusable nuggets" model
rather than bloating one example into a full game?

A good answer separates "already done" from "net-new", names the crate module
each addition should lean on (or be a candidate to harvest into), and leaves a
set of coarse, independently-workable tasks.

## Context

`examples/14_breach.rs` (1360 lines) already has more than the ask assumes. From
reading the source and `docs/2026-07-05-breach-example.md`:

- **Main menu: already exists.** Full `GameState { Menu, Playing, GameOver }`
  state machine (`ui/menu`), a pulsing title, best-score readout, `menu_select`
  SFX, and touch/keyboard dismissal.
- **Mobile: already works.** `TouchpadPlugin` dual virtual sticks + fire button.
  The doc is candid that "an FPS is the hardest genre for touch" and calls it a
  compromise; desktop is primary.
- **Sounds: present but borrowed.** Seven `SfxPlugin` one-shots, all mapped to
  *shared* placeholder files (`launch`/`pickup`/`combo`/`hurt`/`level_up`/
  `menu_select`/`game_over`). No breach-specific audio; `Sfx::Hit` reuses
  `pickup`, `Sfx::EnemyDown` reuses `combo`.
- **Points: a bare kill count.** `Score(u32)` increments once per kill; shown on
  the status bar and persisted as `HighScore<u32>`. No multi-kill/streak logic.
- **Waves: a linear ramp.** `wave_size(n)` and `enemy_speed(wave)` scale count
  and speed; a single `Enemy` type (`ENEMY_HEALTH`, `ENEMY_SPEED_BASE`,
  `ENEMY_DPS`, octahedron mesh). No variety in behaviour or stats.
- **Juice: partial.** `camera/shake` recoil + hit kick, `feedback/screen_flash`
  damage vignette, `feedback/flash` enemy hit-flash, `camera/post` bloom, tracers
  (`helpers/temp`), `mesh/explode` gibs. No hit markers, no kill/combo popups, no
  low-health warning, no muzzle flash entity, no enemy spawn tell.

Crate building blocks already available for the gaps: `scoring/streak` (the
decaying combo counter, backs fruitninja `Combo` and orbit `Streak` -- exactly
the "points+combos" mechanic, and breach does NOT use it yet), `ui/popup` (the
floating "+N" readout), `helpers/temp` (timed pickups/markers), `health`
(pickup heal path), `time/cooldown` (fire-rate the pickup would modify).

So four of the six asks are already shipped in some form; the real net-new value
is **combos, pickups, enemy variety**, plus **polish passes** on the menu, sounds
and juice that already exist.

## Options considered

### How much to build

- **A) One big "breach v2" task.** Bundle every improvement into a single task.
  Pro: one branch, coherent tuning pass. Con: violates the crate's one-concern
  rhythm, hard to review, and mixes cheap polish with genuinely new systems
  (pickups, enemy AI) that each carry their own unknowns. Rejected.
- **B) One task per theme, sized to its novelty (recommended).** Split into
  independent tasks: combos, pickups, enemy variety, sound pass, juice pass,
  menu/mobile polish. Each cites this spike, each is independently workable and
  reviewable, and the genuinely-new ones (pickups, enemies) get their own
  `/plan` pass while the polish ones stay small. Fits how the gallery grew
  (each example is one PR; each harvest is its own task). Chosen.
- **C) Do nothing / close as "already a game".** Defensible for menu+mobile
  (already shipped) but leaves the actual fun gaps (combos, pickups, variety) on
  the table, which is the heart of the ask. Rejected except as the scoping for
  the menu/mobile items, which fold into a single light polish task.

### Where each addition should live

The crate rule is "harvest a reusable nugget only on the second user". Assessed
per item:

- **Combos** -> reuse existing `scoring/streak` + `ui/popup`. No new crate code;
  this is a wiring task. Highest value-to-effort.
- **Pickups** (drop-on-kill or ground spawns granting HP/speed/fire-rate) -> a
  net-new mechanic. Keep the pickup entity + pickup-on-proximity + timed-buff
  logic *game-local* first (like the FP controller was); only flag a possible
  `pickup`/`powerup` crate module as a follow-up if a second example wants it.
  New enough to deserve its own `/plan`.
- **Enemy variety** (e.g. a fast-weak rusher, a slow-tanky brute, maybe a
  ranged/exploder) -> game-local data. The `12_bastion` catalog spike already
  concluded a `SpecCatalog<T>` type should wait for a second user, so keep enemy
  stats as an in-file table/enum, not a JSON catalog. Its own `/plan`.
- **Sound pass** -> generate breach-specific placeholders via
  `scripts/gen-placeholder-sounds.py` (the established pattern from
  `06_fruitninja`), map the currently-borrowed `Sfx` variants to them. Small.
- **Juice pass** -> hit markers, kill/combo popups (`ui/popup`), muzzle flash,
  low-HP vignette pulse, enemy spawn tell. Reuses existing modules; small-to-medium.
- **Menu + mobile polish** -> controls hint on the menu, a difficulty/start
  affordance, and a touch-usability pass (button sizing, aim assist for the
  swarm). Small; folds menu and mobile together since both are already-present
  refinements.

## Recommendation

Adopt **option B**: seed six independent, spike-tagged tatr tasks, each sized to
its novelty, ordered by value-to-effort. Do **not** implement anything now; each
task is picked up later through the normal `/plan` -> `/work` -> `/review` flow.

Priority intent (highest value-to-effort first):

1. **Combos + points** -- pure wiring of the existing `scoring/streak` +
   `ui/popup`; turns the flat kill count into a rewarding multi-kill loop. This
   is also the first real user of `scoring/streak` from a 3D/FPS context.
2. **Ground pickups** -- HP/speed/fire-rate buffs; the biggest new-fun lever and
   a natural companion to combos (reward the aggressive player). Game-local.
3. **Enemy variety** -- 2-3 enemy archetypes with distinct stats/behaviour;
   makes waves read as escalating threats rather than "more of the same".
4. **Juice pass** -- hit markers, kill/combo popups, muzzle flash, low-HP pulse,
   spawn tells. Cheap, high perceived-quality.
5. **Breach sound pass** -- dedicated placeholders replacing the borrowed SFX.
6. **Menu + mobile polish** -- controls hint, start affordance, touch usability
   / aim-assist. Smallest; the base already works.

Guardrails carried into every task, from the crate's accumulated retros:

- Copy Bevy 0.19 UI/light idioms from an existing example, do not improvise
  (font_size, TextLayout, border_radius, AmbientLight bite repeatedly).
- Any *game-driven* state change (new lose/win/level paths) must be covered by a
  headless `App` unit test -- the `AutopilotPlugin` timer cannot prove it (the
  breach melee/lose-condition retro, `docs/retros/20260705-114236-breach-example.md`).
- Pure logic (combo value rule, pickup-buff duration math, enemy stat table,
  wave composition) gets in-module `#[cfg(test)]` tests; assert the *final
  shipped* data if a test embeds it.
- Keep new mechanics game-local; only propose a crate harvest as an explicit
  follow-up task when a mechanic has (or clearly will have) a second user.
- Verify with `cargo clippy --all-targets` (not bare `cargo build`), then a
  headless `BCS_AUTOPILOT` run, then run the example once for real.

## Open questions

- **Pickup delivery model:** drop-on-kill (rewards clearing) vs periodic ground
  spawns (rewards map control) vs both. Lean drop-on-kill to reinforce the combo
  loop; settle during that task's `/plan`.
- **Fire-rate / speed buffs:** timed (expire after N seconds, needs a buff-timer
  component) vs permanent-for-the-run stacking. Timed reads as more arcade-y and
  avoids runaway scaling; confirm at plan time.
- **Enemy count for variety:** 2 vs 3 archetypes for a first cut. Start at 2-3
  (fast rusher + tanky brute, optional ranged) and let the wave composition test
  pin the exact roster (the bastion embedded-catalog-count test lesson applies).
- **Ranged/exploder enemy** would need projectiles or a self-destruct AoE the
  arena has never had -- possibly its own follow-up rather than folded into the
  variety task.
- **Crate harvest candidates** (a `powerup`/pickup module, an enemy-archetype
  helper): deferred to the "second user" rule; not seeded as tasks now.
