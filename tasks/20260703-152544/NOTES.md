# Audio SFX module and fruit ninja sounds

Date: 2026-07-03
Tasks: 20260703-152612 (audio module), 20260703-152619 (wiring),
20260703-152544 (this doc).

## What changed

- New crate module `src/audio` with `SfxPlugin`, a `PlaySfx` event, an
  `SfxMasterVolume` resource, and an `SfxCommandsExt` (`play_sfx` /
  `play_sfx_volume`). Exported through `crate::prelude`.
- `examples/06_fruitninja.rs` now plays a one-shot sound on every gameplay
  event, loaded from `assets/sounds/` into an `SfxAssets` resource.
- Eight placeholder WAVs under `assets/sounds/`, generated by
  `scripts/gen-placeholder-sounds.py`, plus `assets/sounds/README.md` listing
  the assets the user should supply.

## Why a reusable module instead of inlining in the example

Bevy's one-shot audio is already terse
(`commands.spawn((AudioPlayer(h), PlaybackSettings::DESPAWN))`), so the module
is intentionally thin. It earns its place because it gives games three things
the raw spawn does not:

- a fire-and-forget trigger (`PlaySfx` / `play_sfx`) that decouples "I want a
  sound" from the `AudioPlayer` / `PlaybackSettings` mechanics, so gameplay
  systems and observers stay about gameplay;
- a single `SfxMasterVolume` knob for a volume slider or mute, applied to every
  sound in one place;
- prelude ergonomics consistent with the rest of the crate.

This matches the crate's mission (small, composable, game-agnostic building
blocks that almost every game needs). It is deliberately scoped to transient
SFX: no music player, no mixer, no bus routing. Looping background music is a
one-liner with `PlaybackSettings::LOOP` and did not justify widening the
module.

Alternative considered: put all audio logic directly in the example. Rejected
because sound is a near-universal game need and the crate exists precisely to
avoid re-solving those in each game.

## Design notes

- `PlaySfx` is a plain global `Event` (not an `EntityEvent`), mirroring
  `modding::GameEvent`. SFX are not tied to a target entity, so a global
  trigger observed with `On<PlaySfx>` is the right shape.
- The observer spawns
  `(AudioPlayer, PlaybackSettings::DESPAWN.with_volume(Volume::Linear(v)).with_speed(s))`.
  `DESPAWN` mode makes each sound entity self-cleaning, and spawning one entity
  per trigger gives natural polyphony (overlapping sounds).
- Volume is `per_shot * master`, clamped `>= 0`. Speed is clamped to
  `f32::MIN_POSITIVE` so a caller cannot feed rodio a non-positive playback
  rate. The fruit ninja combo chime uses `with_speed` to rise in pitch as the
  chain grows.

## Mock-asset decision (WAV + dev-only decoder)

The library builds on bevy's default features, where the enabled audio decoder
is vorbis (`.ogg`); `.wav` is not decodable by default. Two ways to ship a
runnable placeholder:

1. Generate an `.ogg` to stay on default features. Rejected: producing a valid
   Ogg Vorbis stream from scratch (without ffmpeg/sox, which we do not assume
   are installed) is far more work than a WAV.
2. Ship short `.wav` placeholders and enable bevy's `wav` decoder. Chosen.

The `wav` feature is added to `[dev-dependencies]` only
(`bevy = { version = "0.19.0", features = ["wav"] }`), so it applies to
examples and tests but leaves the library's own default features untouched -
important for a copy-pastable crate that consumers pull in as a dependency.
Vorbis stays default-on, so a user may instead supply `.ogg` assets by changing
only the file extensions in the `SfxAssets` load calls.

`scripts/gen-placeholder-sounds.py` uses only the Python standard library
(`wave`, `struct`, `math`), writes a short quiet sine blip per event (one
distinct pitch each, so events are audible and distinguishable while testing),
and is byte-deterministic. `assets/` is committed (not gitignored) so the
example runs out of the box.

## How the user swaps in real audio

Overwrite each file under `assets/sounds/` with a real sound at the same path
and filename; no code changes are needed. `assets/sounds/README.md` is the
canonical list (file, event, intended character and length). WAV works today;
`.ogg` works too with the one-line extension change noted above.
